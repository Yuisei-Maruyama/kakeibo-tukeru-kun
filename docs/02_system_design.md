## 2. 処理フロー

### 2.1 画像送信時の処理フロー

```
┌──────┐    ┌──────────┐    ┌──────────────┐    ┌─────────┐    ┌──────────┐
│ LINE │───▶│ Webhook  │───▶│ Gemini API   │───▶│Firestore│───▶│ Calendar │
│      │    │ Handler  │    │ (画像解析)   │    │ (保存)  │    │ (登録)   │
└──────┘    └──────────┘    └──────────────┘    └─────────┘    └──────────┘
    ▲                                                              │
    │                                                              │
    └──────────────────── LINE Reply ──────────────────────────────┘
```

**詳細フロー:**

1. ユーザーが LINE で画像を送信
2. LINE Webhook が Cloud Functions を呼び出し
3. 画像を取得し Gemini API で解析
4. 解析結果（日付・カテゴリー・金額・店舗名）を抽出
5. Firestore に支出データを保存
6. Google Calendar にイベントを登録
7. 残高を計算して LINE に返答

### 2.2 定期集計通知の処理フロー

```
┌───────────────┐    ┌──────────────┐    ┌─────────┐    ┌──────┐
│Cloud Scheduler│───▶│ Scheduler    │───▶│Firestore│───▶│ LINE │
│ (15日/月末)   │    │ Handler      │    │ (集計)  │    │通知  │
└───────────────┘    └──────────────┘    └─────────┘    └──────┘
```

### 2.3 月末精算・貯金計算の処理フロー（月末レポートに含む）

```
┌───────────────┐    ┌──────────────┐    ┌─────────┐    ┌──────┐
│Cloud Scheduler│───▶│ Scheduler    │───▶│Firestore│───▶│ LINE │
│ (月末)        │    │ Handler      │    │ (集計)  │    │通知  │
└───────────────┘    └──────────────┘    └─────────┘    └──────┘
```

**月末処理の詳細:**

1. **外食費用 - 貯金額の算出**
   - 各ユーザーの `diningBalance`（外食残金）を取得
   - 貯金額 = `diningBalance`（= 5万円 − 当月外食費用）
   - 各ユーザーの貯金額をレポートに表示

2. **買い物費用 - 精算額の算出**
   - 当月の買い物費用を各ユーザーごとに集計
   - 差額 = |ユーザーAの合計 − ユーザーBの合計|
   - 精算額 = 差額 ÷ 2
   - 少なく払った方 → 多く払った方に精算額を返金

### 2.4 月次残金リセットの処理フロー

```
┌───────────────┐    ┌──────────────┐    ┌─────────┐
│Cloud Scheduler│───▶│ Reset        │───▶│Firestore│
│ (毎月1日)     │    │ Handler      │    │ (更新)  │
└───────────────┘    └──────────────┘    └─────────┘
```

**詳細フロー:**

1. Cloud Scheduler が毎月1日に Reset Handler を呼び出し
2. 全ユーザーの `diningBalance` を 50000 に更新
3. `balanceResetAt` を現在時刻に更新

### 2.5 カレンダー同期の処理フロー

```
┌───────────────┐    ┌──────────────┐    ┌──────────┐    ┌─────────┐
│Cloud Scheduler│───▶│ calendarSync │───▶│ Calendar │───▶│Firestore│
│ (毎日3:00)    │    │ Handler      │    │ (取得)   │    │ (同期)  │
└───────────────┘    └──────────────┘    └──────────┘    └─────────┘
```

**詳細フロー:**

1. Cloud Scheduler が毎日深夜3:00に calendarSync Handler を呼び出し
2. Google Calendar から当月の支出イベントを取得
3. タイトルをパースしてカテゴリー・ユーザー名・金額を抽出
4. Firestore に未登録のイベントのみ追加（重複防止）
5. 外食費用の場合は残高を更新

### 2.6 サブスク自動登録の処理フロー

┌───────────────┐    ┌──────────────┐    ┌─────────┐    ┌──────────┐    ┌──────┐

## 3. データ設計

### 3.1 Firestore コレクション構造

#### users コレクション（自動登録）
```javascript
{
  id: "LINE_USER_ID",           // LINE User ID（自動取得）
  displayName: "田中",           // LINE表示名（自動取得）
  groupId: "GROUP_ID",          // 所属グループID
  isActive: true,               // アクティブ状態
  diningBalance: 50000,         // 外食費用の残金（デフォルト: 5万円）
  balanceResetAt: Timestamp,    // 残金リセット日時（毎月1日にリセット）
  createdAt: Timestamp,         // 初回メッセージ送信時に自動作成
  updatedAt: Timestamp
}
```

**残金管理ルール（外食費用のみ対象）:**
- 各ユーザーは外食費用として月5万円の残金（`diningBalance`）を保有
- 外食費用の支出時のみ、支払ったユーザーの `diningBalance` から差し引く
- 買い物費用は `diningBalance` に影響しない
- 毎月1日に全ユーザーの `diningBalance` を50000にリセット
- **月末時点の `diningBalance` が各ユーザーの「貯金額」となる**

**自動登録の流れ:**
1. ユーザーが初めて画像を送信
2. LINE API から User ID と表示名を取得
3. users コレクションに自動登録
4. 以降は登録済みユーザーとして処理

#### expenses コレクション
```javascript
{
  id: "auto_generated_id",
  userId: "LINE_USER_ID",
  userName: "田中",
  amount: 1280,
  category: "外食費用",        // "外食費用" or "買い物費用"
  storeName: "サイゼリヤ",
  date: Timestamp,           // レシートの日付
  imageUrl: "gs://...",      // 画像保存先（オプション）
  calendarEventId: "xxx",    // Google Calendar イベントID
  createdAt: Timestamp
}
```

**カテゴリー別の処理:**
| カテゴリー | 残金への影響 | 月末処理 |
|------------|-------------|----------|
| 外食費用 | `diningBalance` から差し引く | 残金が貯金となる |
| 買い物費用 | 影響なし | 差額を精算 |

#### settings コレクション
```javascript
{
  id: "global",
  monthlyBudget: 50000,      // 月額予算（1人あたり）
  lineGroupId: "GROUP_ID",   // LINE グループID（初回メッセージ時に自動設定）
  calendarId: "xxx@group.calendar.google.com",
  firstHalfPayerId: "LINE_USER_ID_A",   // 前半担当者（1日〜15日）
  secondHalfPayerId: "LINE_USER_ID_B",  // 後半担当者（16日〜月末）
  updatedAt: Timestamp
}
```

**半月担当制（外食費用のみ）:**
- 1日〜15日: `firstHalfPayerId` のユーザーが外食の支払い担当
- 16日〜月末: `secondHalfPayerId` のユーザーが外食の支払い担当
- 担当者情報は定期レポートに表示される
- ※買い物費用には担当制は適用されない

#### conversations コレクション（対話セッション管理）
```javascript
{
  userId: "LINE_USER_ID",           // セッションオーナー
  groupId: "GROUP_ID",              // グループID
  type: "add_expense",              // 対話タイプ（ConversationType）
  step: "category",                 // 現在のステップ
  data: {                           // 入力データ（ステップごとに蓄積）
    category: "外食費用",
    payerName: "田中",
    amount: 1280
  },
  createdAt: Timestamp,
  expiresAt: Timestamp              // 10分後に自動期限切れ
}
```

**対話タイプ（ConversationType）:**
| タイプ | コマンド | 説明 |
|--------|---------|------|
| `add_expense` | @追加 | 支出登録 |
| `add_schedule` | @予定 | 予定登録 |
| `delete_expense` | @削除 | 支出削除 |
| `initial_setup` | @初期設定 | 担当者設定 |
| `change_settings` | @設定変更 | 担当者変更 |
| `add_subscription` | @サブスク追加 | サブスク登録 |
| `delete_subscription` | @サブスク削除 | サブスク削除 |
| `edit_subscription` | @サブスク変更 | サブスク編集 |
| `add_rent` | @家賃追加 | 家賃登録 |
| `edit_rent` | @家賃変更 | 家賃編集 |

#### subscriptions コレクション（サブスク管理）
```javascript
{
  id: "auto_generated_id",
  groupId: "GROUP_ID",              // グループID
  payerName: "田中",                // 支払い者名
  payerUserId: "LINE_USER_ID",      // 支払い者のLINE User ID
  serviceName: "Netflix",           // サービス名
  amount: 1490,                     // 金額
  startDate: Timestamp,             // 開始日（初回支払日）
  intervalUnit: "month",            // 間隔単位（"week" or "month"）
  intervalValue: 1,                 // 間隔数値（1=毎週/毎月、2=2週/2ヶ月ごと）
  isActive: true,                   // 有効フラグ（削除時はfalse）
  createdAt: Timestamp,
  updatedAt: Timestamp
}
```

**間隔の計算例:**
| intervalUnit | intervalValue | 意味 |
|--------------|---------------|------|
| week | 1 | 毎週 |
| week | 2 | 2週間ごと |
| month | 1 | 毎月 |
| month | 2 | 2ヶ月ごと |

#### rents コレクション（家賃管理）
```javascript
{
  id: "global",                     // 固定ID
  groupId: "GROUP_ID",              // グループID
  payerName: "田中",                // 支払い者名
  payerUserId: "LINE_USER_ID",      // 支払い者のLINE User ID

```

**注意:**
- 家賃は1グループにつき1件のみ（id: "global"）
- 毎月1日に月末日のカレンダーイベントとして登録
- Firestoreのexpensesには保存しない（精算対象外）

## 5. データベース選定理由（Firestore vs Cloud SQL）

### 5.1 比較表

| 観点 | Firestore | Cloud SQL |
|------|-----------|-----------|
| **料金体系** | 従量課金（操作単位） | 常時課金（インスタンス単位） |
| **無料枠** | 読取5万/日、書込2万/日 | なし（最低$7〜10/月） |
| **スケーリング** | 自動（サーバーレス） | 手動設定が必要 |
| **コネクション管理** | 不要 | コネクションプール必要 |
| **Cloud Functions連携** | ネイティブ対応 | VPC Connector必要（追加コスト） |
| **コールドスタート影響** | なし | 接続遅延あり |
| **スキーマ** | 柔軟（スキーマレス） | 厳格（事前定義必須） |
| **JOIN** | 非対応 | 得意 |
| **トランザクション** | ドキュメント単位で限定的 | 強力（ACID完全準拠） |
| **リアルタイム同期** | ネイティブ対応 | 別途実装が必要 |

### 5.2 本プロジェクトで Firestore を選定した理由

#### ✅ コスト最適化
- 月100回程度の利用では**完全に無料枠内**で運用可能
- Cloud SQL は最低でも月$7〜10のインスタンス費用が発生

#### ✅ サーバーレスアーキテクチャとの親和性
```
Cloud Functions + Firestore = フルサーバーレス構成
Cloud Functions + Cloud SQL = コネクション管理の複雑化
```
- Firestore はコネクションプール不要で瞬時に接続
- Cloud SQL は VPC Connector が必要（追加の設定・コスト）

#### ✅ データ構造の適合性
本システムのデータは以下の特徴を持つ：
- 各支出・ユーザー・会話セッションが独立したドキュメント
- 複雑なリレーション（JOIN）が不要
- ドキュメント単位での読み書きが中心

```javascript
// 例: expenses ドキュメント - 独立したJSONオブジェクト
{
  userId: "LINE_USER_ID",
  amount: 1280,
  category: "外食費用",
  storeName: "サイゼリヤ"
}
```

#### ✅ 運用負荷ゼロ
- バックアップ、パッチ適用、スケーリングが自動
- インフラ管理が不要

### 5.3 Cloud SQL が適するケース（参考）

以下のような要件がある場合は Cloud SQL を検討：
- 複雑なリレーショナルクエリ（多数のJOIN）
- 厳格なスキーマ管理が必要
- 既存のRDBスキーマを移行する場合
- 強力なトランザクション保証が必須

## 6. セキュリティ設計

### 6.1 認証・認可

- LINE Webhook: 署名検証（X-Line-Signature）
- Google APIs: サービスアカウント認証
- Firestore: セキュリティルールで保護

### 6.2 シークレット管理

- Secret Manager で以下を管理:
  - LINE_CHANNEL_SECRET
  - LINE_CHANNEL_ACCESS_TOKEN
  - GEMINI_API_KEY
  - GOOGLE_CALENDAR_ID

## 7. エラーハンドリング

| エラー種別 | 対応 |
|------------|------|
| 画像解析失敗 | ユーザーに再送を依頼 |
| カレンダー登録失敗 | リトライ後、失敗通知 |
| DB保存失敗 | リトライ後、管理者に通知 |
| API制限超過 | キューイングで遅延処理 |
| 対話セッションタイムアウト | セッション自動削除（10分） |

## 8. 監視・ログ

- Cloud Logging で全ログを収集
- Cloud Monitoring でアラート設定:
  - エラー率 > 5%
  - レイテンシ > 10秒
